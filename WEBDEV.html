Web-API’s volgens de REST architecturale stijl

In deze opgave ontwerp je een web-API die kan gebruikt worden als webtoepassing in een browser en tegelijkertijd ook als interface voor andere toepassingen. Voor we hieraan beginnen, moeten we ons eerst afvragen wat een web-API precies is en hoe die verschilt van andere soorten API’s. We bespreken hier in het bijzonder web-API’s volgens de Representational State Transfer (REST) architecturale stijl, waarover je meer zal leren in de les over Web-API’s.

Web-API’s bestaan uit resources

Een Application Programming Interface of API biedt – zoals de naam zegt – elementen aan waarmee een toepassing kan ontwikkeld worden. In de Java-context ben je dit al verschillende keren tegengekomen. Bijvoorbeeld, de Swing-bibliotheek laat toe om grafische toepassingen te ontwikkelen en biedt daarvoor een interface aan die bestaat uit klassen zoals JButton en JLabel. Het woord “klasse” toont aan dat een API sterk verweven is met de onderliggende omgeving. Als we bijvoorbeeld kijken naar de Linux kernel-API, zullen we zien dat deze is opgedeeld in functies, aangezien de functiegebaseerde programmeertaal C daar domineert.

Ook het web heeft een eigen type API’s. De essentiële bouwstenen van web-API’s zijn resources. Een resource is een constante, conceptuele relatie waarvan de waarde mogelijk varieert met de tijd. Hieronder vind je voorbeelden van dergelijke resources:

  • de voorpagina van de krant De Standaard
  • het weer van vandaag in Gent
  • het weer op 14 februari in Gent
  • iTunes versie 11.1.3
  • de laatste versie van iTunes

Merk op dat al deze resources verschillend zijn. Zelfs al zou het vandaag 14 februari zijn, dan nog is “het weer van vandaag” een andere resource dan “het weer op 14 februari”. Dat komt omdat een resource gedefinieerd is door haar conceptuele relatie, niet door de waarde van die relatie op een bepaald moment.

Resources op het web worden uniek geïdentificeerd door HTTP-URL’s. Dit wil zeggen dat één URL ten hoogste één resource identificeert, maar meerdere URL’s kunnen dezelfde resource identificeren. Hieronder vind je enkele voorbeelden van URL’s die “het weer van vandaag in Gent” zouden kunnen identificeren:

  • http://example.org/weather/ghent/today
  • http://gent.be/weer/vandaag
  • https://weather.com/forecasts?city=Ghent&date=today

Essentieel is het inzicht dat web-API’s niet bestaan uit methodes of klassen; het is dan ook niet de bedoeling om methodes of klassen na te bootsen over het web. In de praktijk gebeurt dit echter nog vaak: je kan deze API’s herkennen aan URL’s die methodenamen bevatten zoals showProduct of addComment. Dergelijke API’s noemen we “HTTP API’s” omdat ze wel HTTP als protocol gebruiken, maar niet de informatie-architectuur van het web. De API die we in deze opgave zullen bouwen, is een web-API volgens de REST architecturale stijl.

Resources worden voorgesteld als representaties

De conceptuele aard van resources zorgt meteen voor een probleem: hoe kunnen we iets abstracts als “het weer van vandaag in Gent” doorsturen over een internetverbinding? Uiteraard is dit onmogelijk; in plaats daarvan stellen we het weer voor als een HTML- of JSON-document, die we representaties van de resource noemen. Dit is een mogelijke HTML-representatie van het weer van vandaag: 

<div id="main">
  <h1>The weather in Ghent</h1>
  <h2>Today</h2>
  <dl>
    <dt>temperature</dt>
    <dd>20&deg;C</dd>
    <dt>wind speed</dt>
    <dd>5km/h</dd>
  </dl>
  <p><a href="/weather/ghent/tomorrow">Tomorrow's weather</a></p>
</div>

Dit is een JSON-representatie van diezelfde resource: 

{
  "weather": {
    "title": "Today",
    "temp": 20,
    "wind": 5
  },
  "related": [{
    "title": "Tomorrow",
    "href": "http://example.org/weather/ghent/tomorrow"
  }]
}

De reden dat één resource meerdere representaties kan hebben, is dat verschillende cliënten hierdoor van dezelfde resource gebruik kunnen maken. Als mensen in hun browser de URL http://example.org/weather/ghent/today intikken, krijgen ze een gerenderde versie van de HTML-representatie, waarmee ze dan kunnen interageren. Als een webtoepassing in JavaScript dezelfde URL opvraagt, krijgt deze een JSON-versie terug voor verder gebruik. De tekstuele representatie kan bijvoorbeeld handig zijn voor gebruik in de opdrachtprompt, die geen grafische capaciteiten heeft. Opnieuw is dezelfde URL geschikt om deze representatie op te vragen. Het mechanisme dat dit mogelijk maakt heet content negotiation: iedere cliënt geeft aan welke representaties het beste geschikt zijn, en de server probeert deze voorkeur te respecteren door te antwoorden met de best passende representatie die ondersteund wordt. Dit is een belangrijk verschil tussen web-API’s en andere API’s, waarbij het uitgewisselde formaat typisch onveranderlijk blijft.

Het is cruciaal om te begrijpen dat een web-API niet verschilt van de “website” die je bezoekt in een browser. Met andere woorden: de website is de API. Hoewel we in de praktijk vaak gescheiden toegang zien voor mensen en geautomatiseerde cliënten, bestaat hiervoor geen enkele technische noodzaak. In deze opgave zullen we daarom een web-API ontwerpen met één enkele interface die werkt voor zowel mensen (via HTML) als geautomatiseerde cliënten (via JSON).

Resources worden bewerkt met zelf-beschrijvende boodschappen

Web-API’s volgens de REST architecturale stijl gebruiken het HTTP-protocol op een manier waardoor iedere boodschap op zichzelf kan geïnterpreteerd worden. Langs de ene kant betekent dit dat we HTTP stateless gebruiken: iedere boodschap is onafhankelijk van de vorige. Neem bijvoorbeeld het klassieke scenario waarin we zoekresultaten opvragen voor het sleutelwoord “technologie”. We sturen dan een verzoek naar de server: “geef me de eerste pagina zoekresultaten voor ‘technologie’”. Als we ook de tweede pagina willen bekijken, dan sturen we niet “geef me pagina 2”, want deze boodschap is betekenisloos zonder de vorige. In plaats daarvan sturen we “geef me de tweede pagina zoekresultaten voor ‘technologie’”. Hierdoor hoeft de server niet te onthouden wat onze vorige zoekopdracht was of kan zelfs een andere server ons tweede verzoek behandelen.

Langs de andere kant betekenen zelf-beschrijvende boodschappen dat we enkel vooraf gedefinieerde methodes gebruiken. In Java ben je vrij om zelf methodenamen en hun semantiek te kiezen, bijvoorbeeld calculateHeight of createField. Deze namen hebben echter enkel betekenis voor mensen; voor machines zijn dit gewoon tekenreeksen die variëren per toepassing. Het HTTP-protocol definieert een beperkte set methodes zoals GET, PUT, POST en DELETE, waarvan de betekenis vastligt. Elk van deze methoden neemt een URL als argument, en sommigen ook een representatie van een resource. Hun betekenis is als volgt.

GET /books/23
ontvang een representatie van de resource met URL /books/23
DELETE /books/23
verwijder de resource met URL /books/23
PUT /books/23
plaats de gegeven resource op de URL /books/23
POST /books
voer een actie uit met de gegeven resource op de resource met URL /books, bijvoorbeeld een nieuw boek toevoegen aan de verzameling books

Zoals je merkt, is de POST-methode het vaagst gedefinieerd, en dit laat toe om alle acties te voorzien die niet onder de andere methodes vallen. Het is jouw taak als web-API-ontwerper om te definiëren wat de POST-methode betekent voor iedere resource. Buiten het maken van nieuwe resources in de collectie books, kan POST ook worden gebruikt om gerelateerde acties uit te voeren op een bestaande resource. Bijvoorbeeld, een POST-verzoek naar /books/23 kan worden gebruikt om een reactie toe te voegen aan het boek of om het boek aan een favorietenlijst toe te voegen. In zulke gevallen wordt POST gebruikt om een subresource of gerelateerde actie te creëren, terwijl PUT en PATCH worden gebruikt om een bestaande resource direct te wijzigen.

Het is van het grootste belang dat je steeds de HTTP-methodes correct gebruikt, en dat heeft alles te maken met correct resource-ontwerp. Denk niet “viewBook” maar denk “GET /books/23”. Dat geldt ook voor resources die je meestuurt als argument met POST of PUT. Denk niet “POST /users/stefan met body addFriend” maar denk “POST /users/stefan/friendships met body /users/sara”. Dit toont het verschil tussen methode-georiënteerd (een vriend toevoegen) en resource-geörienteerd ontwerp (een vriendschap-resource aanmaken).

De toepassing wordt bestuurd via hypermedia

De laatste unieke eigenschap van de REST architecturale stijl is het gebruik van hypermediabesturingselementen om de staat van de toepassing te veranderen. HTML is de HyperText Markup Language, en bijzonder aan hypermedia is dat hyperlinks en formulieren gebruikt worden om van één resource naar een andere te navigeren. Dankzij dit mechanisme is het inderdaad mogelijk om stateless boodschappen te versturen: als de server de resource “zoekresultaten voor ‘technologie’” genereert, zal deze in de HTML-representatie al de links plaatsen naar de tweede en derde pagina. Op die manier hoef jij als gebruiker niet telkens het volledige verzoek op te stellen, aangezien je gewoon deze links kan activeren. Dat betekent ook dat je als cliënt niet op de hoogte hoeft te zijn van hoe de URL- of resource-structuur van de server eruit ziet. De server hoeft ook niet te onthouden op welke pagina je gebleven bent. Op geen enkel moment hoef je de URL in de adresbalk manueel aan te passen – tenzij je uiteraard naar een andere website wil springen.

Vermits eenzelfde resource verschillende representaties kan aannemen voor verschillende cliënten, is het noodzakelijk om voor elk representatieformaat na te gaan wat de hypermediamogelijkheden zijn. Momenteel is het vaak zo dat enkel in de representaties voor mensen (HTML) hypermediabesturingselementen voorzien worden. Dat betekent dat de URL’s in automatische toepassingen hardgecodeerd moeten zijn. Meer flexibele toepassingen worden echter mogelijk als we ook hyperlinks aanbieden aan deze automatische toepassingen: op die manier kunnen deze gewoon links volgen zoals wij dit doen, in plaats van ieder verzoek opnieuw te construeren. Je ziet dit in het JSON-voorbeeld, waar links opgenomen werden in de JSON-representatie. Een toepassing die deze representatie ontvangt, kan de link naar het weer van morgen volgen, zonder dat kennis nodig is over de URL-structuur van de server. Dit is opnieuw typisch aan REST web-API’s.

Een raamwerk voor webontwikkeling

Voor ontwikkeling van een webtoepassing zullen we gebruikmaken van een raamwerk voor de programmeertaal JavaScript in de Node.js-omgeving. Dit laat toe om op een efficiënte manier webtoepassingen aan te maken en uit te breiden, en maakt het makkelijker om de REST architecturale stijl te volgen.

Node.js en MVC-raamwerken

Node.js is een omgeving die toelaat van JavaScript code uit te voeren op een server. Express en fastify zijn raamwerken om webtoepassingen te ontwikkelen in Node.js.

Er bestaan veel verschillende raamwerken om web-API’s te bouwen. In verschillende situaties kunnen verschillende raamwerken aangewezen zijn vanwege de verschillende architecturale eigenschappen van deze raamwerken. In deze opgave ben je vrij om een raamwerk te kiezen dat gebruik maakt van JavaScript/TypeScript in de Node.js-omgeving. We geven enkele mogelijkheden mee, maar je bent vrij om een ander raamwerk te kiezen, zolang het een volwaardig MVC-raamwerk is die toelaat om REST API’s te bouwen. Als je niet zeker bent over de keuze van een raamwerk, bespreek dit met de begeleiders. Het belangrijkste van dit practicum is namelijk dat je de principes van REST web-API’s begrijpt en kan toepassen, die van toepassing zijn overheen verschillende MVC-raamwerken.

Als je gebruik maakt van de Express generator (of bijvoorbeeld fastify-cli en @fastify/view voor fastify) is het eenvoudig om een applicatie te maken volgens het Model-View-Controller-patroon (MVC), zoals je dit ook kent in de context van grafische gebruikerstoepassingen. Dit ziet er meestal uit als volgt.

Een model bestaat uit een combinatie van een datatype-definitie (een lijst met velden en types) en een klasse die het gedrag in programmacode bevat. In REST-termen zou je kunnen stellen dat een model overeenkomt met een resource-type; een instantiatie van een model is een resource.

Een blogpost kan gedefinieerd worden door de velden title (string), body (text), en published (boolean), en een klasse die vereist dat body tenminste 300 woorden moet bevatten voor de blogpost gepubliceerd mag worden.

Een view zet één of meerdere modellen om in een serializeerbaar formaat, zoals HTML of JSON. Views worden gegenereerd aan de hand van sjablonen en/of programmacode. Dit komt overeen met een representatie in de REST-terminologie.

Een HTML-representatie van een lijst van blogposts wordt gegenereerd door alle titels op te vragen en deze vervolgens in een HTML-sjabloon te plaatsen.

Een controller voert bewerkingen uit op modellen als antwoord op een HTTP-verzoek. Controllers zorgen ervoor dat de HTTP-methodes het gewenste effect hebben voor een bepaald resource-type.

Als antwoord op een verzoek “POST /articles” maakt een controller een nieuwe blogpost aan en plaatst een statusbericht op de pagina van de auteur.

Een uitgebreidere beschrijving van Node.js en het raamwerk Express kan je vinden in de eerste 2 secties van deze handleiding. Tijdens de ontwikkeling van een MVC-project wissel je doorgaans in hoog tempo af tussen programmeren aan modellen en controllers, en het ontwerpen van views. Hierdoor zal je snel de voordelen merken van een dynamische programmeertaal. Indien je kiest voor een ander MVC-raamwerk (zoals fastify, Koa, NestJS, Hono, ...) kan je de Express-specifieke zaken overslaan, en in de plaats de handleiding van jouw gekozen raamwerk raadplegen (zoals bijvoorbeeld fastify's Getting Started handleiding)

Er zijn veel manieren om te programmeren in Node.js, gaande van een standaard tekst verwerker tot uitgebreide IDEs. Een vrij goede IDE is WebStorm van JetBrains. Die is betalend, maar als student kan je gratis gebruik maken van de volledige Jetbrains catalogus via je UGent-adres. Een andere veelgebruikte optie is VS Code, een gratis IDE van Microsoft. Het is geen verplichting om deze te gebruiken, je bent vrij om je eigen ontwikkelomgeving te kiezen.

IDE's zoals Webstorm en VS Code bieden uitgebreide debuggers aan (Webstorm, VS Code) Deze debuggers zijn nuttig om stap-voor-stap door je code te lopen tijdens uitvoeringstijd, om te zien hoe de logica in de praktijk loopt, en wat effectieve waarden zijn van je variabelen. Bij het uitvoeren van JavaScript in je browser, kunnen debuggers in browsers zoals Chrome ook nuttig zijn.

Taken

Kennismaking met het gekozen raamwerk

Voor deze taak bouw je een voorbeeld met de handleiding Express web framework. Koos je voor een ander raamwerk, dan kun je de Express-specifieke zaken negeren en aanvullend de Getting Started handleiding van het gekozen raamwerk doornemen, zoals de Fastify Getting Started handleiding. Deze handleidingen zijn vrij uitgebreid, dus als je het gevoel hebt dat je vertrouwd genoeg bent met de werkwijze, kan je al overstappen naar de volgende taak. Het is echter belangrijk dat je de structuur van het resultaat begrijpt, want die verwachten we in je oplossing voor de volgende taak.

Bepaalde zaken uit de handleiding dien je anders aan te pakken voor de volgende taken:

  • Gebruik de native JavaScript Promises en async/await syntax in plaats van de async library.
  • Gebruik je eigen MongoDB instantie op je server in plaats van een online instantie.
  • Gebruik voor je route-bestanden een gelijkaardige opdeling als bij de models/controllers; plaats ze niet allemaal in 1 file.
  • Plaats niet alle views in 1 folder; groepeer ze in subfolders.
  • Indien je gebruik maakt van externe JavaScript bestanden voor de views, gebruik package.json en node_modules in plaats van online verwijzingen.
  • Vermijd codeduplicatie. In de tutorial komen bepaalde patronen code meermaals opnieuw voor; dit zien we liever niet.
  • Gebruik HTTP DELETE-berichten om iets te verwijderen.
  • Gebruik HTTP PUT als je de informatie van een object volledig overschrijft, gebruik HTTP PATCH als je dit gedeeltelijk doet.
  • Zorg ervoor dat geen objecten zonder referenties achterblijven: als je een resource verwijdert, moeten ook alle ervan afhankelijke resources verdwijnen.

Deze taak hoef je niet in te dienen.

Een webtoepassing voor audioboeken implementeren

In dit tweede deel bouwen we een toepassing om een bibliotheek van audioboeken te beheren. In deze bibliotheek kunnen audioboeken toegevoegd, aangepast, en verwijderd worden. Op deze audioboeken kunnen gebruikers hun persoonlijke afspeelpositie ingeven, en beoordelingen toevoegen.

We hebben minimaal volgende info nodig

  • gebruiker
    • naam
    • e-mailadres
  • audioboek genre
    • naam
    • beschrijving
  • audioboek
    • naam
    • auteurs
    • genres
    • beschrijving
    • datum van publicatie
    • aankoop link
    • duur (milliseconden)
  • beoordeling
    • gebruiker
    • audioboek
    • beschrijving
    • score
  • afspeelpositie
    • gebruiker
    • audioboek
    • positie (milliseconden)

Op het profiel van een gebruiker dient een lijst te staan van alle audioboeken die de gebruiker aan het beluisteren is, en de reeds beluisterde audioboeken.

Gebruikers kunnen enkel beoordelingen plaatsen op audioboeken die ze reeds (gedeeltelijk) afgespeeld hebben. Bouw ook de nodige validatie in, zodat namen niet leeg kunnen zijn, e-mailadressen geldig zijn, etc.

Het verwijderen van audioboeken moet er enkel voor zorgen dat het audioboek niet meer publiek zichtbaar is. De eigenlijke gegevens moeten bewaard blijven. Dit om toe te laten de "verwijderde" audioboeken nog bekeken kunnen worden.

Alle andere gegevens (gebruikers, genres, beoordelingen, afspeelposities) moeten wel effectief verwijderbaar zijn. Hou hierbij rekening met het aanpassen van gelinkte gegevens, zoals bijvoorbeeld de referenties naar genres binnen audioboeken.

Het is niet de bedoeling om gebruikersaccounts en authenticatie te implementeren, hoewel dat uiteraard wel nodig zou zijn om een dergelijke toepassing uiteindelijk te lanceren. We willen echter bewust focussen op het correct ontwerp van resources. Je mag er dus van uitgaan dat iedereen alle gegevens kan aanpassen, zonder eerst te moeten inloggen. Hierbij kan je bijvoorbeeld bij elke actie een gebruikersveld toevoegen in de formulieren waar nodig, zodat de server de gebruiker en zijn rol kan verifiëren en de vereiste logica kan toepassen.

De webtoepassing uitbreiden met JSON-representaties

In de vorige taak heb je HTML-representaties gemaakt die kunnen bediend worden door mensen via browsers. Voor deze taak voeg je JSON-representaties toe die door machines kunnen geconsumeerd worden.

Let erop dat alle data beschikbaar is in deze representaties, en zorg er ook voor dat de links beschikbaar zijn, zoals de REST achitecturale stijl vraagt. Je kan JSON-representaties testen met de toepassing curl. Gebruik dezelfde URL als voor HTML, maar stel de optie --header in op “Accept: application/json”. Of beter nog, maak je eigen hypermedia-type aan dat een specialisatie is van JSON.

Je toepassing uitrollen op een server

Je krijgt een lege Ubuntu-server ter beschikking waarop je je toepassing dient te installeren en publiek beschikbaar te maken. De login-gegevens worden je bezorgd via mail. Deze server heeft geen firewall, dus zorg ervoor dat je geen onveilige services publiek toegankelijk maakt. Indien je een eigen firewall instelt, zorg ervoor dat je niet de SSH-poort blokkeert, anders kun je niet meer op de server.

Je dient volgende zaken te voorzien:

  • Installatie van Node.js en npm.
  • Installatie van de NGINX-webserver als publieke server.
  • Je eigen webtoepassing automatisch laten starten als de server start.
  • NGINX configureren zodat je webtoepassing beschikbaar is in de root van het hoofddomein van je server. Bijvoorbeeld, als je de domeinnaam groep01.webdev.stud.atlantis.UGent.be kreeg, dan is je applicatie beschikbaar op https://groep01.webdev.stud.atlantis.UGent.be/.
  • NGINX configureren zodat andere hostnames een 404 geven. Bijvoorbeeld, als we proberen je server te bereiken via http://example.org/, dan moet de server antwoorden met een 404.
  • Een SSL-certificaat aanvragen via Let's Encrypt en dit certificaat instellen via NGINX. Je site moet dus beschikbaar zijn via HTTPS met een geldig certificaat. Opgelet: vraag zo snel mogelijk een certificaat aan gezien Let's Encrypt een limiet heeft van 20 certificaten per subdomein per week en er meer dan 20 groepen zijn.
  • Gewone HTTP-URLs moeten permanent redirecten naar de HTTPS-URL. Bijvoorbeeld: http://groep01.webdev.stud.atlantis.UGent.be/foo/bar moet redirecten naar https://groep01.webdev.stud.atlantis.UGent.be/foo/bar
  • Statische assets (in de /assets/-map, mag een andere naam hebben) moeten gecached worden door NGINX voor een uur.
