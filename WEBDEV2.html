Cliënten van REST web-API’s

In de vorige opgave maakte je kennis met REST web-API’s. Een centraal principe van REST is dat de server slechts één API aanbiedt die door verschillende types cliënten kan gebruikt worden. In deze opgave implementeren we webtoepassingen die deze API gebruiken.

Een belangrijke uitdaging in deze opgave is om te leren hoe je cliënten voor een REST API schrijft. Hierbij is het belangrijk dat je de REST-principes in het achterhoofd houdt, in het bijzonder dat de toepassing wordt bestuurd via hypermedia. Dit betekent dat er geen hard-gecodeerde URL's in je broncode mogen staan, met uitzondering van één begin-URL. De volledige communicatie tussen jouw cliënt en de server moet gebeuren via hypermedia, dus door het volgen van links.

Om er zeker van te zijn dat de API die je gebruikt de REST-principes correct implementeert, gebruiken we voor deze opgave een voorbeeldimplementatie van opgave 1. Deze API is beschikbaar via https://groep40.webdev.stud.atlantis.ugent.be/. Zoals je kan zien, worden enkel de JSON-representaties aangeboden. Je kan deze resources bekijken door de URL’s die je tegenkomt in de adresbalk van je browser te plakken. De toepassingen die je ontwikkelt, zullen ook gebruik maken van deze URL’s. Aangezien deze API voor iedereen openstaat, resetten we de inhoud iedere dag. Gewijzigde data wordt dus maximaal 24 uur bewaard. Experimenteer alvast met deze API, bijvoorbeeld via de browser en/of met curl.

Webtoepassingen ontwikkelen

Om een webtoepassing te ontwikkelen gebruik je HTML voor de basisstructuur, CSS voor de opmaak op verschillende schermgroottes en JavaScript voor de interactie met de reeds bestaande web-API.

Qua CSS verwachten we dat je toepassing responsief is; dat wil zeggen dat deze de besturingselementen zodanig aanpast dat deze vlot leesbaar zijn op kleine en grote schermen. Je kan dit doen met het sleutelwoord @media. De meeste browsers hebben ook een optie om websites te bekijken in verschillende formaten, zodat je dit makkelijk kan testen.

Aangezien jullie allemaal gebruik maken van dezelfde API kan het zijn dat je op hetzelfde moment als iemand anders een aanpassing probeert te doen. Om te voorkomen dat je per ongeluk een aanpassing overschrijft die je niet eens hebt gezien, kan je in HTTP gebruik maken van zogenaamde preconditions. Daarmee kan je expliciet zeggen welke versie van een document je wil aanpassen. De API vereist dat je hier gebruik van maakt, dus lees zeker bij over de ETag en If-Match headers.

Je toepassing kan single-page zijn (met één HTML-bestand), of multi-page (met dan typisch een HTML-bestand per type resource).

API handleiding

Data uitlezen van de API doe je met GET. Er zijn 3 soorten pagina's:

  • De startpagina. Deze bevat links naar pagina's voor alle specifieke model pagina's.
  • Een model pagina. Deze bevat een oplijsting van alle object pagina's die overeenkomen met dit model.
  • Een object pagina. Deze bevat alle informatie over een specifiek object.

Nieuwe objecten aanmaken doe je met een POST naar een model pagina. Elke model pagina bevat een veld requiredFields. Deze duidt aan welke velden verwacht worden als je een nieuw object aanmaakt. Er is ook een optionalFields veld dat aanduidt welke extra velden je mag meesturen.

Objecten aanpassen doe je met PUT of PATCH. Met PUT moet je een volledig nieuw object doorsturen en worden dezelfde verwacht als bij een POST. Met PATCH kan je een object doorsturen dat niet alle velden bevat. Alleen de aanwezige velden worden vervangen. Ten slotte kan je een object verwijderen met DELETE.

De API antwoordt met HTTP status codes om aan te duiden of een aanvraag succesvol was of niet. Indien er een probleem is, kan je de statuscode en bijhorende foutmelding gebruiken om uit te vissen wat er nu precies foutloopt.

Om objecten aan te maken of aan te passen moet je je authoriseren. Dit doe je door een Authorization header mee te sturen met je request met waarde Bearer $TOKEN. Elke groep heeft een eigen token die je zal opgestuurd krijgen.

Een responsieve webtoepassing bouwen

Waar de toepassing in Opgave 1 nagenoeg volledig op de server werd uitgevoerd, plaatsen we in deze taak de verantwoordelijkheid voor de gebruikersinteractie op de cliënt.

De bedoeling is om een webtoepassing te bouwen die toelaat om de publieke API te bedienen. De functionaliteit van deze toepassing is gelijkaardig aan de toepassing uit de vorige opgave, met het verschil dat de onderliggende functionaliteit wordt aangeboden door een externe API. Concreet verwachten we dat een gebruiker deze handelingen kan uitvoeren:

  • gebruikers weergeven, aanmaken, bewerken en verwijderen
  • audioboeken weergeven, aanmaken, bewerken en verwijderen. In tegenstelling tot bij de vorige opgave kunnen audioboeken wel permanent verwijderd worden.
  • genres weergeven, aanmaken, bewerken en verwijderen
  • afspeelposities weergeven, aanmaken, bewerken en verwijderen
  • beoordelingen weergeven, aanmaken, bewerken en verwijderen

Hou er rekening mee dat er in de API ook velden zijn die niet rechtstreeks door de client kunnen aangepast worden. Zo worden er bijvoorbeeld bij een gebruiker de corresponderende wagens meegestuurd, maar om dit aan te passen zou je een nieuwe wagen moeten aanmaken, of een bestaande verwijderen.

Gebruik een van de volgende JavaScript-raamwerken:

  • React
  • Vue.js
  • Angular

Om je toepassing te testen, kan je een lokale webserver draaien zoals serve. Uit veiligheidsoverwegingen van je browser werken namelijk niet alle functies als je een HTML-bestand rechtstreeks van het bestandssysteem laadt.

Let erop dat je toepassing werkt op kleine en grote schermen; gebruik responsive CSS om elementen te herschalen waar nodig. Alle menu-elementen die tegen elkaar gedrukt worden op een klein scherm is geen responsief design. Zorg er in de plaats voor dat de menu-elementen verdwijnen achter een knop op een klein scherm.

Hou rekening met de niet-functionele vereiste dat hypermedia je toepassing moet aandrijven: slechts één URL mag hardgecodeerd zijn, de overige resources moeten gevonden worden door links te volgen. Als je ergens in je code API URL's split of zelf samenstelt, ben je waarschijnlijk fout bezig.

De nadruk ligt niet op de gebruikersinterface, maar des te meer op de communicatie met de API. Je mag veronderstellen dat de toepassing constant een internetverbinding ter beschikking heeft. Precieze inputvalidatie is niet nodig voor deze Opgave; vertrouw op de foutafhandeling van de server, maar geef wel een duidelijke indicatie van slagen of falen op de pagina. De foutmeldingen van de API mag daarvoor gebruikt worden zonder aanpassing.

Zorg ervoor dat alle pagina's naar elkaar kunnen linken. Zo moet het mogelijk zijn om op de pagina van een wagen te klikken op de verkoper, om zo op de pagina van die gebruiker terecht te komen. Er mogen ook geen URL’s of ID’s zichtbaar zijn voor de gebruiker. Bijvoorbeeld: op die pagina van de wagen moet de naam van de eigenaar zichtbaar zijn en niet de URL.

Er is geen limiet op hoeveel data er in de API kan zitten, dus zorg er voor dat je website ook vlot kan omgaan met een lijst van duizenden elementen. Maak gebruik van asynchrone acties die de pagina aanpassen terwijl de data wordt ingeladen.

Hou rekening met de verschillende foutcodes waar de server mee antwoordt en zorg dat je website die correct kan afhandelen. Zo kan het bijvoorbeeld gebeuren dat een resource wordt aangepast of verwijderd door iemand anders terwijl een gebruiker op een pagina zit om die zelf aan te passen.

De webtoepassing bouwen met een ander raamwerk

Voor de tweede taak, bouw je een toepassing die aan dezelfde vereisten voldoet als de eerste, maar dan met een ander raamwerk. Je mag kiezen uit de bovenstaande, of een eigen raamwerk naar keuze (of zelfs zonder raamwerk, als je dat wil). Zorg wel dat je raamwerk echt verschillend zijn, Next.js is bijvoorbeeld niet verschillend van React.

Hergebruik zoveel mogelijk tussen de raamwerken, om jezelf werk uit te sparen. De bedoeling is vooral om de developer experience van verschillende raamwerken te ervaren.

Indienen

Werk samen op UGent GitHub en zorg ervoor dat je oplossing enkel zichtbaar is voor je eigen team. Maak een apart GitHub-project aan voor elke website. Dien je oplossing in ten laatste op 17 april via Ufora als een zip-bestand met daarin:

  • je eerste webtoepassing in de submap web1, zonder node_modules,
    met tevens je .git-map hiervan (≥10 commits)
    en een statische versie van je website toegankelijk via web1/dist/,
  • je tweede webtoepassing in de submap web2, zonder node_modules,
    met tevens je .git-map hiervan (≥10 commits)
    en een statische versie van je website toegankelijk via web2/dist/,

Zorg er voor dat je de statische versie van je websites indient in de dist map! De sites zullen getest worden door een lokale webserver naar die dist map te laten wijzen. Er zullen geen angular/react/vue/etc. commando's uitgevoerd worden om de website werkende te krijgen.

Beoordelingscriteria

De verbetering zal gebeuren door in de dist folder van elke website serve --single uit te voeren. Je kan dit dus zelf eenvoudig testen. Er zal voor elke site gekeken worden naar onder andere de volgende criteria:

  • Kunnen objecten bekeken/aangemaakt/aangepast/verwijderd worden?
  • Worden foutmeldingen correct afgehandeld en weergegeven op de pagina?
  • Is er voor elk object een detailpagina die correct linkt naar de relevante objecten?
  • Wordt er gebruik gemaakt van responsive design. Verdwijnen de menu knoppen bijvoorbeeld op een kleiner scherm.
  • Werkt de site correct in verschillende browsers?
  • Maakt de site correct gebruik van de JSON-paden en werkt die nog als de URL-structuur verandert?
  • Stuurt de applicatie de correcte precondition headers mee.
  • Er wordt niet gekeken naar visueel design, maar een minimum aan gebruiker experience wordt wel verwacht. Klik dus wat rond op je website en zie of die het gebruiksgemak heeft dat je zelf zou verwachten.
